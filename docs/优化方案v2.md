## Phase 1: 前端补齐（释放已有后端能力）

**目标**：后端已实现但前端未暴露的 5 个模块全部补上 UI，评分从 22/50 提升到 30/50。

### 1.1 数据导入页面

**当前状态**：后端有 CSV/支付宝/微信导入 API，无前端页面

**实现内容**：
- 新建 `ImportPage.tsx`，加入侧边栏导航
- 文件拖拽上传组件（Ant Design Upload + Dragger）
- 格式选择：支付宝 / 微信 / 通用 CSV，带格式说明图示
- 通用 CSV 模式：列映射配置界面（下拉选择每列对应字段）
- 导入预览表格：解析后展示前 20 条，用户确认后入库
- 去重提示：显示跳过的重复记录数和详情
- 导入结果摘要：成功 N 条 / 跳过 N 条 / 失败 N 条
- 导入历史列表（按 importedId 分组显示历史导入批次）

**Gemini Review 补充**：
- CSV 导入需考虑用户上传文件的编码多样性（UTF-8/GBK/GB18030），应提供编码选择或自动检测
- 日期格式（YYYY-MM-DD / MM/DD/YYYY / DD.MM.YYYY）和金额格式（1,234.56 / 1.234,56）需支持预设或自定义正则
- 去重策略应允许用户在预览阶段调整判断标准（如仅按金额+日期，或加上描述）

**对标参考**：
- Firefly III Data Importer 的列映射 + 预览流程
- ezBookkeeping 的格式选择 + 编码自动检测

### 1.2 标签管理页面

**当前状态**：后端 CRUD + 交易关联 API 完整，无前端

**实现内容**：
- 标签管理入口（可作为设置子页或独立页面）
- 标签列表：彩色标签卡片展示，支持编辑名称/颜色、删除
- 新建标签：名称 + 颜色选择器（预设色板 + 自定义 hex）
- 交易表单集成：在 TransactionPage 的记账表单中添加标签多选下拉
- 交易列表集成：在交易行中显示关联标签（彩色小标签）
- 按标签筛选：交易列表增加标签筛选条件
- 报表集成：收支分析增加"按标签"维度的统计视图

**对标参考**：
- Firefly III 的标签管理 + 交易内标签选择
- ezBookkeeping 的标签组分组

### 1.3 规则管理页面

**当前状态**：后端规则引擎完整（triggers + actions + AND/OR），无前端

**实现内容**：
- 新建 `RulePage.tsx`，加入侧边栏导航
- 规则列表：卡片式展示，显示标题、条件摘要、动作摘要、启用/禁用开关
- 规则创建/编辑表单：
  - 标题输入
  - 匹配模式切换（全部满足 AND / 任一满足 OR）
  - 触发条件组：动态添加/删除行，每行选择条件类型 + 输入值
    - 描述包含 → 文本输入
    - 金额大于/小于 → 数字输入
  - 执行动作组：动态添加/删除行，每行选择动作类型 + 输入值
    - 设置分类 → 分类下拉
    - 添加标签 → 标签下拉
    - 设置描述 → 文本输入
- 规则排序：拖拽调整执行优先级
- 手动应用：选择交易范围批量应用规则
- 规则测试：输入样例交易数据，预览匹配结果

**后端增强**：
- 新增 trigger 类型：`account_is`（账户匹配）、`description_regex`（正则匹配）、`amount_between`（金额范围）
- 新增 action 类型：`set_account`（设置账户）、`convert_type`（转换交易类型）
- 导入时自动触发规则：importService 创建交易后调用 ruleService.applyRules

**对标参考**：
- Firefly III 的规则管理界面（条件/动作列表 + 优先级排序）

## Phase 2: 多币种完整集成

**目标**：Schema 已支持，前端 + 后端逻辑完整打通。

### 2.1 币种管理

- 内置 ISO 4217 币种表（180+ 币种，含代码/名称/符号/小数位）
- 用户默认币种设置（User 模型增加 defaultCurrency 字段）
- 常用币种快捷选择：CNY / USD / EUR / JPY / GBP / HKD

### 2.2 账户币种

- 创建账户时选择币种（默认为用户默认币种）
- 账户列表按币种分组或显示币种标识
- 不同币种账户余额独立展示

### 2.3 外币交易

- 记账表单支持外币金额输入：选择外币 → 输入外币金额 → 输入汇率 → 自动计算本币金额
- 或：输入本币金额 + 外币金额，自动计算汇率
- 交易列表同时显示本币和外币金额

### 2.4 汇率管理

- 后端增加 ExchangeRate 模型（或直接用现有 exchangeRate 字段）
- 自动汇率获取服务：接入免费汇率 API（如 exchangerate-api.com 或 open.er-api.com）
- 定时更新（每日一次）
- 手动汇率录入/覆盖
- 汇率历史查看

### 2.5 报表多币种

- 资产负债表：按币种分组显示，总计按默认币种换算
- 收支分析：外币交易按交易日汇率折算为默认币种
- Dashboard：净资产按默认币种统一展示

**Gemini Review 补充**：
- 多币种换算必须统一舍入规则（如四舍五入到小数点后两位），避免微小差额累积
- 外币交易必须记录交易发生日的历史汇率（而非实时汇率），保证报表数据可追溯
- 后端存储交易时应同时保存当时的汇率快照

**对标参考**：Firefly III 的 foreign_amount 模式、ezBookkeeping 的多数据源汇率

## Phase 3: 报表与分析增强

**目标**：从基础报表升级到可与 Firefly III 竞争的分析能力。

### 3.1 趋势分析

- 月度收支趋势折线图（最近 12 个月）
- 分类支出趋势（选定分类的月度变化）
- 资产净值趋势（每月末净资产快照）
- 支持按周/月/季/年切换时间粒度

### 3.2 同比环比

- 收入/支出同比增长率（与去年同期对比）
- 收入/支出环比增长率（与上月对比）
- 可视化：柱状图 + 增长率标注

### 3.3 现金流量表

- 经营活动现金流（收入 - 支出）
- 投资活动现金流（如果有投资账户）
- 筹资活动现金流（借贷相关）
- 期初/期末现金余额

### 3.4 试算平衡表 UI

- 当前 API 已有，前端增加 Tab 展示
- 所有账户借贷汇总，验证会计等式

### 3.5 分类排行

- Top 10 支出分类排行（柱状图）
- Top 10 收入来源排行
- 异常检测：标记本月显著高于平均的分类

### 3.6 账户余额变动

- 选定账户的余额变动历史（折线图）
- 按日/周/月粒度
- 大额变动标注

### 3.7 报表导出

- 导出为 PDF（使用 html2canvas + jsPDF）
- 导出为图片（PNG）
- 定期报表邮件发送（可选，需邮件服务配置）

**Gemini Review 补充**：
- 数据量增长后复杂报表需后端聚合计算，前端应考虑数据抽样和虚拟滚动
- 报表 UI 应预留自定义筛选灵活性（时间范围、分组方式、多维度交叉）

**对标参考**：
- Firefly III 的 Chart 模块（折线/柱状/饼图组合）
- Beancount+Fava 的 bean-query 自由查询
- GnuCash 的现金流量表

## Phase 4: 导入能力增强

**目标**：从 3 种格式扩展到 10+ 种，并加入 AI 和自动化能力。

### 4.1 新增导入格式

- **OFX/QFX**：国际银行标准格式（使用 `ofx-js` 或自行解析 SGML）
- **QIF**：Quicken 导入格式
- **京东金融**：CSV 解析（参考 ezBookkeeping 的 jdcom 解析器）
- **随手记**：CSV/XLS/XLSX 三种格式（参考 ezBookkeeping 的 feidee 解析器）
- **银行 PDF 账单**：解析信用卡月账单 PDF（使用 `unpdf` 提取文本 + 正则匹配）
  > ⚠️ **Gemini 风险提醒**：银行 PDF 账单格式千变万化，即使同一银行不同时期格式也不同，正则维护成本极高，可能是"无底洞"。建议：(1) 先仅支持 2-3 家主流银行的固定格式；(2) 引入社区驱动的"解析器模板"机制，让用户贡献和维护不同银行的解析规则；(3) 优先级放低，优先做 CSV/邮件导入

### 4.2 导入预览与编辑

- 解析后展示完整表格，每行可编辑
- 自动匹配已有账户/分类（模糊匹配）
- 未匹配项高亮，用户手动选择或创建
- 批量修改分类/账户
- 选择性导入（勾选需要的行）

### 4.3 导入规则集成

- 导入后自动应用规则引擎
- 预览阶段显示规则将如何修改每条交易
- 支持"从本次导入创建规则"快捷操作

### 4.4 邮箱账单自动导入（差异化功能）

- IMAP 邮箱连接配置（支持 Gmail/QQ/163/Outlook）
- 定时扫描新邮件（可配置频率）
- 银行邮件识别规则：
  - 按发件人匹配（如 `creditcard@notice.cmbchina.com`）
  - 按主题关键词匹配（如"信用卡账单"、"交易提醒"）
- 邮件内容解析：
  - HTML 邮件正文提取（交易金额、商户名、日期）
  - 附件 CSV/PDF 自动下载并解析
- 解析结果暂存待确认队列
- 用户确认后批量导入

**Gemini Review 补充**：
- 不同邮件服务商对 IMAP 连接的限制和行为不同，需充分测试 QQ/163/Gmail 的兼容性
- 某些服务商可能对短时间内大量连接或下载附件有频率限制
- 邮箱密码必须 AES 加密存储，绝不明文

## Phase 5: 自动化与智能化

**目标**：从手动记账向半自动/全自动演进。

### 5.1 规则引擎增强

- 新增 trigger 类型：
  - `description_regex`：正则表达式匹配
  - `account_is`：指定源/目标账户
  - `amount_between`：金额范围（min-max）
  - `date_is_weekday` / `date_is_weekend`：按星期匹配
  - `tag_has`：已有某标签
- 新增 action 类型：
  - `set_account`：修改账户
  - `convert_type`：转换交易类型（支出→转账）
  - `set_tag`：覆盖（非追加）标签
  - `append_description`：追加描述文本
  - `set_budget`：关联预算
- 规则链：一条规则的输出作为下一条的输入
- 规则模板库：预设常用规则（如"外卖自动归入餐饮"、"滴滴自动归入交通"）

**Gemini Review 补充**：
- 规则数量多时注意正则表达式性能，编写不当可能导致 ReDoS
- 可考虑对规则进行索引或优化匹配顺序（先匹配简单条件再匹配正则）

### 5.3 Webhook 事件通知

- 交易创建/更新/删除时触发 Webhook

## Phase 7: 安全与部署

**目标**：达到生产可用的安全和部署标准。

### 7.1 安全增强

- **2FA (TOTP)**：Google Authenticator / Authy 支持
  - 后端：`otpauth` 或 `speakeasy` 库
  - 前端：二维码展示 + 验证码输入
- **WebAuthn**：硬件安全密钥支持（YubiKey 等）
- **会话管理**：活跃会话列表 + 远程登出
- **审计日志**：记录所有增删改操作（谁/什么时间/改了什么）
- **数据加密**：敏感字段加密存储（账户余额等）
- **密码策略**：复杂度要求、过期提醒

### 7.2 Docker 部署

- 后端 Dockerfile（Node.js 多阶段构建）
- 前端 Dockerfile（Nginx 静态服务）
- docker-compose.yml（一键启动前端+后端+数据库）
- 数据卷挂载（SQLite 文件持久化）
- 环境变量配置
- 健康检查端点
- 支持 Unraid/Synology NAS 部署

### 7.3 数据库升级路径

- 支持 PostgreSQL（生产环境）
- Prisma migrate 正式迁移流程（替代 db push）
- 自动备份脚本（定时备份 SQLite 文件）

## Phase 8: 高级记账功能

**目标**：追赶 Beancount/GnuCash 的专业会计能力。

### 8.1 科目体系增强

- 增加 Equity（所有者权益）账户类型

### 8.2 对账（Reconciliation）

- 选择账户 + 银行对账单余额

- 自动计算差额
- 生成调整分录（对账差异）
- 标记已对账交易（locked，不可修改）

### 8.3 余额断言

- 参考 Beancount 的 `balance` 指令

- 断言失败时告警（数据可能有误）

### 8.4 投资追踪

- 投资账户类型（股票/基金/加密货币）

- 实现/未实现盈亏计算
- 市场价格获取（接入行情 API）
- 投资组合概览：持仓分布饼图 + 收益率

### 8.5 交易附件

- 上传收据/发票照片或 PDF

- 缩略图预览
- 全文搜索附件内容（OCR 提取文本后索引）

### 8.6 分类层级

- 两级或无限级分类（如 餐饮 > 外卖 / 餐饮 > 堂食）

- 按层级汇总报表（一级分类汇总 + 展开看二级）

## Gemini Peer Review 总结

### Round 1: Gemini 2.5 Flash（2026-04-08）

**整体评价**：方案详尽周密，**所有核心选型均获认同，无推翻项。**

| Phase | 评价 | 关键建议 |
|-------|------|---------|
| Phase 1 | ✅ | CSV 编码/日期/金额格式多样性；去重策略可调 |
| Phase 2 | ✅ | 换算舍入规则统一；交易记录当日历史汇率 |
| Phase 3 | ✅ | 后端聚合性能；报表自定义筛选灵活性 |
| Phase 4 | ⚠️ | **银行 PDF 解析是"无底洞"**，建议先支持 2-3 家或社区驱动 |
| Phase 5 | ✅ | 正则 ReDoS 性能风险 |
| Phase 6 | ✅ | PWA 离线同步冲突解决是难点 |

### Round 2: Gemini 3.1 Pro（2026-04-08）

**整体评价**：方案质量极高，但部分依赖在自托管场景下过重。**核心理念：压榨浏览器原生能力和 SQLite 内置功能，减少外部依赖。**

#### 依赖替换建议

| 原方案 | Gemini 3.1 建议 | 理由 |
|--------|----------------|------|
| `@react-pdf/renderer` (PDF 导出) | **❌ 改用 CSS `@media print` + `window.print()`** | 零依赖，复用现有 ECharts/Antd 组件，只需几行打印样式 |
| `sharp` (图片处理) | **⚠️ 改用前端 Canvas 压缩** | sharp 的 libvips 在 Docker/NAS 部署频繁出兼容问题 |
| `fuse.js` (前端模糊搜索) | **❌ 删掉，全用后端 SQLite FTS5** | 前端只需防抖调 API，不需内存全量搜索 |
| `natural` (NLP 智能分类) | **⚠️ 改用 SQLite FTS5 BM25** | 直接查最相似历史交易取其分类，零依赖，中文效果更好 |
| `exceljs` (Excel 解析) | **⚠️ 改用 `read-excel-file`** | 更轻量，exceljs 多年未大更新 |
| `node-cron` (定时任务) | **⚠️ 改用 DB 轮询表** | 进程重启丢任务，基于 SQLite 记录 last_run_time 更可靠 |

#### Phase 优先级调整建议

| 原顺序 | Gemini 3.1 建议 | 理由 |
|--------|----------------|------|
| 1 → 2 → 3 → 4 → 5 → 6 → 7 | **1 → 6 → 7 → 2 → 3 → 4 → 5** | 移动端体验（记账 80% 在手机上）和 Docker 部署（数据安全）应大幅提前 |

### 最高风险项

1. **银行 PDF 账单解析**（Phase 4.1）— 维护成本可能远超预期，建议低优先级或社区驱动
2. **PWA 离线同步冲突**（Phase 6.2）— 同一交易离线+在线同时修改的冲突解决需要仔细设计
3. **IMAP 邮箱兼容性**（Phase 4.4）— 不同服务商的限制和行为差异需充分测试
4. **sharp 跨平台部署**（Phase 8.5）— Docker Alpine/NAS 环境 libvips 兼容性问题频发
