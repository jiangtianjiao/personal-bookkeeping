# 个人记账项目评估报告

> 生成日期：2026-04-08

---

## 一、功能实用性评估

### 已实现的核心能力

| 功能 | 状态 | 评价 |
|------|------|------|
| 复式记账法 | ✅ 已实现 | 借贷必平衡验证、四类科目（资产/负债/收入/支出）、余额按科目类型正确计算 |
| 快速记账 | ✅ 已实现 | 收入/支出/转账三种简化模式，自动生成借贷分录 |
| 手动分录 | ✅ 已实现 | 支持多借多贷复杂分录 |
| 财务报表 | ✅ 已实现 | 资产负债表、收支分析（含饼图）、试算平衡表 |
| Dashboard | ✅ 已实现 | 净资产、月度收支、最近交易 |
| 分类体系 | ✅ 已实现 | 17个预设分类 + 自定义分类 |
| 用户认证 | ✅ 已实现 | JWT 登录注册 |

### 缺失的关键能力

| 功能 | 状态 | 影响 |
|------|------|------|
| 邮箱账单读取 | ❌ 未实现 | 无法自动解析银行/信用卡邮件账单 |
| 数据导入/导出 | ❌ 未实现 | 无 CSV/Excel/OFX/QIF 导入，每笔交易必须手动录入 |
| 金额精度 | ⚠️ 用了 Float | 浮点数计算会导致精度丢失，记账软件必须用 Decimal |
| 多币种支持 | ❌ 未实现 | 无货币字段，无法处理外币消费 |
| 周期性交易 | ❌ 未实现 | 房租、工资等固定支出/收入无法自动生成 |
| 预算管理 | ❌ 未实现 | 无法设定月度/年度预算并追踪执行情况 |
| 标签/项目维度 | ❌ 未实现 | 只有分类维度，无法按项目（如旅行）聚合查看 |
| 移动端体验 | ⚠️ 仅响应式 | 不是原生 App 或 PWA，移动端快速记账体验不佳 |

### 与主流方案对比

| 能力 | 本项目 | Beancount | GnuCash | 随手记 |
|------|--------|-----------|---------|--------|
| 复式记账 | ✅ | ✅ | ✅ | ❌ |
| 银行数据导入 | ❌ | ✅ (插件) | ✅ (OFX) | ✅ |
| 邮件账单解析 | ❌ | ✅ (插件) | ❌ | ❌ |
| 多币种 | ❌ | ✅ | ✅ | ✅ |
| 预算 | ❌ | ✅ | ✅ | ✅ |
| 数据导出 | ❌ | ✅ (纯文本) | ✅ | ✅ |
| 移动端 | ⚠️ | ❌ | ❌ | ✅ |
| 自动化规则 | ❌ | ✅ | ❌ | ❌ |

### 功能评估总结

**当前状态**：一个架构正确的复式记账 MVP，会计逻辑没有硬伤。

**核心短板**：**数据录入效率太低**。没有导入、没有邮件解析、没有周期性交易、没有移动端快速记账——所有数据都要在网页上手动一笔一笔录入，实际使用中会迅速被放弃。

**功能优先级建议**：
1. Float → Decimal — 基础正确性问题
2. CSV/银行流水导入 — 实用性从 0 到 1
3. 周期性交易 — 减少重复劳动
4. 多币种 — 个人记账基本需求
5. 邮件账单解析 — 进一步自动化

## 二、代码框架评估

### 后端核心问题

| 严重度 | 问题 | 位置 |
|--------|------|------|
| **Critical** | **缺失 Service 层** — 所有业务逻辑堆在 Controller 里，`quickEntry` 单个方法 200 行，不可测试 | 全部 Controller |
| **Critical** | **N+1 查询** — 账户余额逐个计算，Dashboard 请求 = 3+N 条 SQL | `accountController.ts:28-37`, `reportController.ts` 多处 |
| **Critical** | **`computeAccountBalance` 复制粘贴** — 完全相同的函数出现在两个 Controller | `accountController.ts:259`, `reportController.ts:9` |
| **Critical** | **Float 存金额** — 精度丢失，用 epsilon 比较是治标不治本 | `schema.prisma:88` |
| **High** | **`types/index.ts` 全文件死代码** — 定义了 schema 中不存在的字段，从未被引用 | `types/index.ts` |
| **High** | **无输入验证库** — 手动 `if (!field)` 检查，不验证 email 格式/密码强度/日期合法性 | 全部 Controller |
| **High** | **错误处理泛化** — 所有异常一律 catch 返回 500，Prisma 结构化错误被吞掉 | 全部 Controller |
| **High** | **`initializeUserData` 无事务** — 逐个 `await create`，中途失败用户半初始化 | `authController.ts:246` |
| **High** | **初始化数据重复定义** — `authController.ts` 和 `seed.ts` 各维护一份相同数据 | 两个文件 |
| **Medium** | **冗余 `!userId` 检查** — middleware 已保证，Controller 中重复 20+ 次 | 全部 Controller |
| **Medium** | **JWT Secret 硬编码 fallback**，CORS 全开放，无 rate limit | `auth.ts:4`, `index.ts:17` |
| **Medium** | **响应格式不统一** — 有的返回数组，有的包 `{ data }`，有的包 `{ account }` | 各 Controller |

### 前端核心问题

| 严重度 | 问题 | 位置 |
|--------|------|------|
| **Critical** | **状态管理矛盾** — `AuthContext` 是 `authStore` 的纯透传，两种访问路径共存 | `AuthContext.tsx`, `authStore.ts` |
| **Critical** | **`appStore.ts` 整文件死代码** — 89 行，从未被任何组件引用 | `store/appStore.ts` |
| **Critical** | **Token 双重存储** — `zustand/persist` 写 `auth-storage`，又手动写 `auth_token`，API 层读后者 | `authStore.ts:40`, `api.ts:20` |
| **High** | **`TransactionPage` 651 行 God Component** — 表单 + 列表 + 弹窗 + 列定义全部堆在一个文件 | `TransactionPage.tsx` |
| **High** | **`formatAmount` 重复定义 3 次**、`getTransactionType` 重复 2 次、`isMobile` 重复 5 次 | 各 Page 文件 |
| **High** | **无数据获取抽象** — 每个页面重复 `loading/setLoading/try-catch-finally` 模式 | 各 Page 文件 |
| **Medium** | **大量 `any` 类型** — 20 处，主要在错误处理和 Table 列渲染 | 多处 |
| **Medium** | **401 处理粗暴** — 直接 `window.location.href` 跳转，不清 store，多请求竞争 | `api.ts:42` |
| **Medium** | **无路由级代码分割** — ECharts 等大依赖打进主包 | `App.tsx:7-13` |
| **Medium** | **颜色硬编码** — `#ef4444` 出现 12 次、`#10b981` 9 次，CSS 变量定义了但不用 | 各 TSX 文件 |
| **Low** | **零测试**、无 a11y、Vite 配置缺路径别名 | — |

### 架构评分

| 维度 | 后端 | 前端 |
|------|------|------|
| 分层架构 | 2/5 (缺 Service 层) | 2/5 (God Component) |
| 类型安全 | 2/5 (大量 any + 死类型) | 3/5 (大量 any 但主类型正确) |
| 代码复用 | 2/5 (复制粘贴) | 2/5 (工具函数重复) |
| 查询/性能 | 1/5 (N+1) | 3/5 (基本可用) |
| 可测试性 | 1/5 (零测试 + 不可测) | 1/5 (零测试) |
| 安全性 | 2/5 | 3/5 |

### 推荐修复优先级

1. **后端引入 Service 层** — 提取 `BalanceService`、`TransactionService`、`UserService`
2. **修复 N+1** — 用 `groupBy` 或 raw SQL 聚合余额
3. **前端删除死代码** — 删 `AuthContext`、删 `appStore.ts`、修 token 双重存储
4. **提取公共函数** — `formatAmount`/`getTransactionType`/`useIsMobile`
5. **拆分 TransactionPage** — 至少拆为 Form + List + EntryDetail
6. **引入 Zod 做输入验证** — 替代散落的 if 检查
7. **Float → Decimal** — 财务数据精度问题

### 对本项目的路径建议

**路径 A：直接用现有开源方案** — 如果目标是"好用的记账工具"，Firefly III 或 ezBookkeeping Docker 一键部署即可。

**路径 B：参考优秀项目重构** — 以 Actual Budget 的架构为参考（同技术栈），以 Firefly III 的功能集为路线图，借鉴 ezBookkeeping 的微信/支付宝导入。

**路径 C：基于 Beancount 生态构建** — 用 Beancount 做核心引擎（数据存储 + 验证 + 查询），自己写 React 前端，核心会计逻辑由经过十年验证的引擎保证正确性。
